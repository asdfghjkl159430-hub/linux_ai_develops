# 摘要

本项目围绕《Linux 系统管理与Shell编程》课程所推荐的 **「AI Agent Skills」** 方向，设计并实现一套在 Linux 下可运行的运维自动化原型：**以自然语言意图为输入，经 Agent 路由映射到受控 Shell 流水线（runbook），对磁盘、日志与进程健康进行巡检与诊断输出**。系统强调 **Shell 脚本、grep/sed/awk 文本处理、管道与重定向、进程与日志分析** 等课程核心技术，辅以 **Python 白名单命令执行** 与 **Git 过程管理**，并给出 **crontab** 定时巡检示例，符合课程对「能运行、能解释、能测试、能维护」的要求。

**关键词**：Linux；Shell；Agent Skills；grep/awk；自动化运维；日志审计

# 第一章 项目背景

## 1.1 行业与教学背景

随着云主机与容器化环境的普及，Linux 运维工作大量体现为「命令组合 + 排障经验」的重复劳动。近年来 AI Agent 技术快速发展，课程规范鼓励学生在 Linux 场景下探索 **Agent 工具调用** 与 **Shell 自动化编排**，而非单纯调用聊天接口。

## 1.2 问题陈述

传统脚本虽可完成巡检，但缺少「意图层」：不同同学编写的脚本入口参数各异，复用成本高。Agent Skills 的思想是将 **操作步骤与边界** 以结构化文档固化，并由轻量路由层选择执行路径。本项目以课程可控复杂度实现该思想：**关键词路由 + runbook 白名单命令**。

## 1.3 建设目标

1. 提供 2～3 个核心运维能力：磁盘、日志、进程。  
2. 所有能力可在无图形界面下运行，便于 SSH 环境演示。  
3. 输出可追踪：路由与 runbook 结果写入 `logs/`。  
4. 提供完整文档与测试，支撑答辩与互评。

# 第三章 系统设计

## 3.1 总体架构

系统分为四层：**交互层（shell 命令行）**、**路由层（agent-router）**、**执行层（runbooks）**、**支撑层（lib + skills 文档）**。数据流为：用户字符串 → 归一化小写 → 关键词正则匹配 → 选择脚本 → `bash` 子进程执行 → 标准输出经 `tee` 持久化。

## 3.2 模块设计

### 3.2.1 磁盘巡检模块

采集 `df` 人类可读输出与 inode 使用率；若存在 `findmnt` 则补充挂载关系；使用 `awk` 对容量列进行阈值判断并输出 WARN/INFO。

### 3.2.2 日志审计模块

对默认或用户给定路径执行尾部截取与关键字过滤；计数采用 `grep -c` 与 `awk` 聚合。对不可读系统日志提供跳过与 sudo 提升两种路径。

### 3.2.3 进程健康模块

聚合 `uptime`、`ps` Top 进程、`vmstat` 可选输出、`systemctl --failed` 可选输出，用于观察瞬时压力与失败服务。

### 3.2.4 Agent 路由模块

采用 **多关键词析取** 的简化分类器，避免引入重型 NLP 依赖，符合课程对 Linux 基础的聚焦。

## 3.3 接口与约定

- 路由成功执行 runbook 后以 runbook 退出码为准。  
- 路由无法识别意图：退出码 3。  
- 日志目录默认 `logs/`，尝试放宽权限以适配实验室多用户同路径写入需求。

# 第四章 功能实现

## 4.0 Web 交互界面（Flask）

新增 `web/app.py` 提供浏览器入口：用户在页面输入自然语言后，前端通过 `fetch('/api/run')` 提交 JSON；服务端使用 `subprocess.run` 以 **参数列表** 方式调用 `bin/agent-router.sh`，禁止 shell 元字符拼接，从而在不扩大攻击面的前提下完成「Agent 接收 → 执行 → 回显」闭环。该设计与 CLI 共用同一执行路径，便于对照测试与答辩演示。

## 4.1 开发环境与目录结构

项目根目录：`/opt/kcsj/linux-agent-skills`。`bin/` 存放入口；`runbooks/` 存放 Shell 流水线；`skills/` 存放 Agent 可读规范；`tests/` 存放自动化测试脚本。

## 4.2 关键技术实现说明

### 4.2.1 管道与文本处理

`log_audit.sh` 中通过 `tail | grep | awk` 组合完成「截取—过滤—聚合」三阶段处理，是课程 grep/awk 知识点的集中体现。

### 4.2.2 可选 sudo 读文件

通过 `common.sh` 中 `agent_maybe_sudo_cat` / `agent_maybe_sudo_tail` 封装：优先普通读；失败则在 `AGENT_USE_SUDO=1` 且 `sudo -n` 可用时提升。该设计兼顾 **「读取权限给最高」** 的课设演示诉求与基本安全边界。

### 4.2.3 Python 白名单执行

`lib/safe_exec.py` 使用 `subprocess.run` 并校验 argv 前缀，演示如何在脚本语言层再做一道防线，防止将用户输入直接拼接入 shell -c。

## 4.3 与 Agent Skills 文档的对应关系

每个 `skills/*/SKILL.md` 描述：触发条件、执行步骤、安全注意、与路由关键词对齐，便于在 Cursor 等工具中作为技能包阅读，亦便于答辩展示「规格与实现一致」。

# 附录 B：Shell 编程要点复盘（扩写）

## B.1 `set -euo pipefail` 的作用

`-e` 在命令返回非零时终止脚本，避免错误被静默忽略；`-u` 防止未定义变量拼写错误；`pipefail` 使管道中任一命令失败即可被捕获。对运维脚本而言，该组合能显著提升失败可见性，但亦要求对「grep 未匹配」等返回码 1 的场景使用 `|| true` 或 `if` 包裹。

## B.2 文本处理三件套的分工

`grep` 擅长按模式过滤行；`sed` 适合流式替换与截取；`awk` 适合字段切片与聚合统计。本项目中 `awk` 主要用于对日志关键字计数按前缀聚合，属于轻量级统计而非复杂 DSL。

## B.3 日志读取权限模型

Linux 系统日志常归属 `adm`/`syslog` 用户组，普通开发者账号无读权限。课设环境若需「读取权限给最高」，推荐路径为：

1. 实验机为演示账号配置 **免密 sudo** 且仅允许 `cat`/`tail` 针对特定路径（sudoers 精细化）；  
2. 或将日志复制到可读目录后再分析；  
3. 或使用 `journalctl --user` 读取用户级日志作为替代演示数据。

本项目实现第 1 类路径为可选开关，避免脚本默认提权。

## B.4 与 cron 的结合思路

`crontab` 适合固定周期巡检。示例中将路由入口作为 cron job，可把结果追加到按日滚动的日志文件，实现最小成本的「无人值守巡检」。注意 cron 环境变量较干净，应在 crontab 中显式写入 `PATH` 与项目绝对路径。

A：课程未强制；Docker 为加分项。本项目聚焦 Shell 与系统命令，后续可将 runbook 容器化。
